# AgentRadar-Stat 项目需求文档

## AI Agent 开源生态趋势挖掘、质量评分与智能选题决策平台

## 1. 项目名称

**AgentRadar-Stat：AI Agent 开源生态趋势挖掘、质量评分与智能选题决策平台**

副标题：**基于 GitHub 公开仓库数据、统计建模、PyTorch 表格神经网络与 DeepSeek 多智能体的开源项目决策平台**

项目简称：**AgentRadar-Stat**

项目定位关键词：

**500 个 GitHub 公开仓库、多模型评分、聚类画像、PyTorch 表格神经网络、个性化推荐、DeepSeek Agent 解释、HTML/PDF 自动报告、Streamlit 交互系统。**

------

## 2. 项目定位

AgentRadar-Stat 是一个面向 AI Agent、LLM、RAG、Coding Agent、MCP、Vibe Coding、Workflow Agent、Data Agent 等开源生态的 Python 数据产品。系统以真实 GitHub 公开仓库数据为主数据源，通过 GitHub Search API、GitHub REST API、README 文本、topics、仓库元数据与人工指定 seed 仓库补充，构建 AI Agent 相关开源项目样本，并在此基础上完成数据采集、数据清洗、特征工程、指标体系构建、综合评分、聚类画像、机器学习预测、PyTorch 表格神经网络扩展、用户画像个性化推荐、DeepSeek 多智能体解释、HTML/PDF 项目报告与 Streamlit 交互系统。

本项目不是普通的 GitHub 排行榜可视化，也不是单一 RAG 问答系统，更不是单独展示某一种模型算法，而是将“如何从 GitHub 上快速增长、类型复杂的 AI Agent 开源生态中，识别出值得学习、复现、二次开发或持续关注的项目”转化为一个可量化、可解释、可交互、可复现的 Python 数据科学项目。

项目最终形成两个核心展示形式：

第一，**AgentRadar-Stat HTML/PDF 项目报告**。报告用于完整呈现项目背景、数据采集、指标体系、趋势洞察、评分排行、聚类画像、模型预测、PyTorch 扩展、个性化推荐、DeepSeek Agent 解释、项目亮点、局限展望与附录输出。

第二，**AgentRadar-Stat Streamlit 交互系统**。Streamlit 用于实时用户画像选择、项目筛选、单项目评分构成查看、个性化推荐解释、聚类地图交互、模型结果查看、DeepSeek Agent 卡片解释和报告导出。

两者同源于同一套数据、模型、评分体系、推荐逻辑和 Agent 输出。报告 HTML/PDF 侧重于最终成果展示，Streamlit App 侧重于交互式探索和个性化使用。

------

## 3. 项目真实应用场景

随着 AI Agent、Coding Agent、MCP、RAG、Workflow Agent 与 Vibe Coding 工具快速发展，GitHub 上每天都会出现大量相关开源项目。单纯依靠 Stars、Trending 排名、项目标题或 README 第一印象，难以判断一个项目是否真正具有学习价值、工程质量、技术潜力、维护活跃度、文档完整度和复现可行性。

对于课程项目复现者而言，他们需要判断哪些项目能在有限时间内复现并展示。对于开发者而言，他们需要识别哪些项目值得二次开发或学习工程结构。对于技术社区观察者而言，他们需要理解 AI Agent 开源生态的主题结构、项目类型和发展趋势。对于开源项目维护者而言，他们需要从社区参与、Issue 压力、文档质量和风险暴露角度观察项目状态。

AgentRadar-Stat 正是面向这一真实需求，将 GitHub 公开数据从“简单排行榜”升级为“统计建模 + 质量评分 + 项目画像 + 个性化推荐 + Agent 解释”的智能开源项目决策系统。

## 4. 项目核心问题

本项目围绕以下问题展开：

1. 当前 GitHub 上 AI Agent 相关开源项目主要集中在哪些技术方向？
2. 哪些项目只是短期热度较高，哪些项目具有持续维护潜力？
3. 哪些项目文档完整、依赖清晰、示例充分，适合学习、复现或二次开发？
4. 哪些项目存在维护停滞、Issue 压力高、README 不完整、依赖复杂或运行门槛较高等风险？
5. 能否基于 GitHub 公开仓库数据构建一个多维项目潜力评分体系，而不是只按 Stars 排序？
6. 能否通过聚类画像识别不同类型的 AI Agent 开源项目？
7. 能否通过监督学习任务检验评分体系的可解释性，并观察哪些原始特征能够解释项目潜力？
8. 能否通过 PyTorch 表格神经网络扩展传统机器学习模型，体现深度学习在结构化开源仓库数据上的应用？
9. 能否根据用户身份、目标、编程水平、偏好语言、偏好主题、运行条件、风险偏好和输出偏好生成个性化项目推荐？
10. 能否使用 DeepSeek Agent 将字段、图表和模型结果转化为自然语言解释、风险提示、复现路线和 Codex Prompt？
11. 能否将最终成果以 HTML/PDF 报告和 Streamlit App 的形式完整呈现，形成可展示、可交互、可复现、可上传 GitHub 的完整数据产品？

## 5. 目标用户

第一类用户是希望快速了解 AI Agent 开源生态的学习者。他们希望通过可视化和统计分析快速掌握 AI Agent、MCP、RAG、Coding Agent、Workflow Agent 等方向的项目分布，而不是逐个浏览仓库。

第二类用户是希望选择学习项目或复现项目的开发者。他们关心项目热度、文档质量、依赖复杂度、维护状态、复现可行性和潜在风险，希望筛选出真正值得投入时间的项目。

第三类用户是寻找课程项目、科研入门、创新实践或开源贡献方向的学习者。他们希望系统不仅给出项目排名，还能提供推荐理由、复现路线、风险提示和可执行开发 Prompt。

第四类用户是技术社区观察者和开源项目维护者。他们希望从语言生态、主题分布、社区活跃、Issue 压力、文档质量和维护风险等角度比较项目状态。

第五类用户是 AI 工具与软件工程研究者。他们关注 AI Agent 与开源协作之间的关系，希望通过公开数据观察项目活跃度、主题演化和生态结构。

## 6. 项目最终目标

本项目最终目标是形成一个端到端 Python 数据产品，具体包括：

1. 使用真实 GitHub API 采集 AI Agent 相关开源项目数据；
2. 使用扩展关键词和 seed 仓库补充构建高相关项目样本；
3. 支持本地 sample 数据作为 API 异常时的兜底机制，但展示层默认以 GitHub 公开仓库数据为主；
4. 对仓库元数据、README 文本、topics、Stars、Forks、Watchers、Issues、更新时间、许可证、项目规模等字段进行清洗和标准化；
5. 构建趋势热度、活跃维护、社区参与、文档质量、技术创新、复现可行和风险惩罚等七维指标；
6. 构建综合潜力分、风险分、推荐等级和个性化匹配分；
7. 使用描述统计、相关分析、分布分析和主题分布观察 AI Agent 开源生态；
8. 使用 PCA、多算法聚类比较和聚类画像识别不同项目类型；
9. 设计 Task A / Task B 两个监督学习任务，并进行防泄漏建模；
10. 使用逻辑回归、随机森林、极端随机树、梯度提升树、校准模型和基线模型进行模型比较；
11. 使用 PyTorch 表格宽深神经网络（TabularWideDeepNet）完成深度学习扩展；
12. 构建用户画像体系与个性化推荐机制；
13. 使用 DeepSeek API 实现多智能体解释，包括采集策略、数据质量、主题识别、评分解释、项目顾问、报告生成和事实审查；
14. 使用 Plotly 等工具生成交互式图表；
15. 使用 Streamlit 构建多页面交互系统；
16. 自动生成 HTML 项目报告和 PDF 项目报告；
17. 提供完整 README、AGENTS.md、运行说明、开发日志、测试脚本和 GitHub 仓库交付材料；
18. 最终上传 GitHub，形成可复现、可展示、可继续扩展的完整项目仓库。

## 7. 数据来源设计

### 7.1 主数据源：GitHub API

项目主流程以 GitHub API 为基准数据源。核心包括：

1. **GitHub Search Repositories API**
   按 AI Agent、RAG、MCP、Coding Agent、Workflow Agent、Vibe Coding 等关键词搜索相关仓库。
2. **GitHub Repository API**
   获取仓库元数据，包括 repo_id、owner、repo_name、full_name、repo_url、description、language、topics、stars_total、forks_total、watchers_total、open_issues_count、created_at、updated_at、pushed_at、license、default_branch、archived、size 等字段。
3. **GitHub Contents / README API**
   获取 README 文本，用于识别安装说明、快速开始、Demo、示例、依赖说明、API Key、GPU、Docker、Streamlit、Gradio 等复现相关信号。
4. **seed repositories 补充机制**
   对 Dify、AutoGPT、CrewAI、AutoGen、LangGraph、OpenHands、MetaGPT、Semantic Kernel、MCP Servers 等高相关项目设置 seed 仓库，通过 GitHub API 拉取真实元数据，并与关键词搜索结果统一去重。
5. **GitHub Trending 页面**
   可作为趋势信息补充来源，但不替代 GitHub API 主流程。

### 7.2 兜底数据源：本地 sample 数据

本地 sample 数据只作为异常兜底机制。当 GitHub API 不可用、网络异常、Token 缺失、限流或课堂展示环境无法访问外网时，系统可启用本地 sample 数据保证流程可运行。

兜底机制必须满足：

1. 不影响主流程以 GitHub API 为基准；
2. 不将 sample 数据冒充真实 API 数据；
3. 在日志和 run_summary 中保留 fallback 记录；
4. 在正式展示层不突出调试标签，避免影响项目展示完整性；
5. 课堂展示和最终报告默认使用 GitHub 公开仓库数据结果。

### 7.3 DeepSeek API 解释源

项目使用 DeepSeek API 支持多智能体解释，DeepSeek API Key 通过 `.env` 或环境变量读取，不得硬编码在代码中。

建议环境变量包括：

```env
GITHUB_TOKEN=
DEEPSEEK_API_KEY=
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-pro
USE_DEEPSEEK_AGENTS=true
OFFLINE_FALLBACK=true
MAX_REPOS=500
PER_KEYWORD_LIMIT=50
MAX_README_REPOS=300
MAX_AGENT_REPOS=40
```

DeepSeek Agent 的输出用于解释、推荐、风险提示和报告生成辅助，不替代原始字段、统计图表和模型指标。

## 8. 采集关键词与样本规模

最终关键词应覆盖 AI Agent 开源生态的主要方向，包括但不限于：

ai-agent、agentic-ai、llm-agent、multi-agent、autonomous-agent、coding-agent、code-agent、software-agent、dev-agent、rag-agent、retrieval-augmented-generation、llmops、mcp、model-context-protocol、workflow-agent、research-agent、browser-agent、data-agent、data-analysis-agent、autogpt、babyagi、langchain-agent、crewai、autogen、semantic-kernel、openai-agent、local-llm-agent、computer-use、ai-coding、vibe-coding、agent-framework 等。

采集策略包括：

1. 多关键词搜索；
2. 多排序策略搜索，包括 stars、updated、forks 和 best match；
3. seed 仓库补充；
4. full_name 唯一去重；
5. 相关性过滤；
6. README 抓取数量控制；
7. Agent 分析数量控制。

在当前最终版本中，项目默认保留 **500 个 GitHub 公开仓库** 作为评分、聚类、模型训练和推荐的主样本。候选仓库数量、去重后数量与最终入模数量应根据每次运行的 `run_summary.json` 动态更新，不在需求文档中过度固定具体中间值。

## 10. 数据清洗与特征工程

### 10.1 数据清洗

系统需要完成以下清洗步骤：

1. 读取 GitHub API 采集结果；
2. 构造仓库唯一标识 owner/repo；
3. 删除重复仓库；
4. 标准化仓库名称、URL、description、topics 和 README；
5. 对数值字段进行类型转换；
6. 对时间字段进行格式转换；
7. 计算项目年龄和距离最近更新时间；
8. 对 Stars、Forks、Issues 等偏态变量进行 log1p 转换；
9. 标记 archived 或明显无效仓库；
10. 清理 README 中的 markdown 标记、链接、代码块和特殊字符；
11. 记录 API 状态与异常原因；
12. 输出 cleaned_repos.csv，作为后续分析基础。

### 10.2 特征工程

系统构建七类特征。

第一类是趋势热度特征，包括 log_stars、log_forks、stars_per_day、stars_this_week、trending_flag。

第二类是活跃维护特征，包括 days_since_update、pushed_recently_flag、project_age_days、activity_score。

第三类是社区参与特征，包括 forks_per_star、watchers_total、issue_pressure、community_score。

第四类是文档质量特征，包括 readme_length、has_install_section、has_quickstart、has_demo、has_example、has_requirements、documentation_score。

第五类是技术创新与 AI Agent 相关性特征，包括 agent_keyword_count、agent_relevance_score、topic_label、coding_agent_flag、rag_flag、mcp_flag、multimodal_flag、data_agent_flag。

第六类是复现可行性特征，包括 mentions_cpu、mentions_gpu、mentions_docker、mentions_api_key、dependency_complexity_signal、feasibility_score。

第七类是风险惩罚特征，包括 long_time_no_update_flag、high_issue_pressure_flag、low_doc_quality_flag、too_large_repo_flag、api_key_required_flag、gpu_required_flag、risk_score。

## 11. 综合评分体系

系统构建六个正向子评分和一个风险惩罚项：

1. Trend Score：趋势热度分；
2. Activity Score：活跃维护分；
3. Community Score：社区参与分；
4. Documentation Score：文档质量分；
5. Innovation Score：技术创新分；
6. Feasibility Score：复现可行分；
7. Risk Score：风险分。

默认综合评分公式为：

[
FinalPotentialScore =
0.25 \times TrendScore

- 0.20 \times ActivityScore
- 0.15 \times CommunityScore
- 0.15 \times DocumentationScore
- 0.15 \times InnovationScore
- 0.10 \times FeasibilityScore

- 0.10 \times RiskScore
  ]

所有子评分应使用 MinMaxScaler、分位数标准化或稳健标准化，避免单一 Stars 指标支配全部结果。

推荐等级划分为：

A：强烈推荐；
B：值得关注；
C：普通参考；
D：暂不推荐。

Streamlit 支持基于用户画像调整推荐权重，使同一项目对不同用户呈现不同的个性化优先级。

## 13. 聚类画像模块

聚类画像用于把仓库划分为不同项目类型，帮助理解项目在潜力、风险和工程成熟度上的差异。聚类结果用于解释项目结构，不作为监督学习标签。

系统应实现：

1. PCA 降维；
2. KMeans 聚类；
3. Gaussian Mixture 聚类；
4. 层次凝聚聚类；
5. DBSCAN 密度聚类；
6. 聚类指标比较，包括轮廓系数、DB 指数、选择分和稳定性指标；
7. 最终聚类类型命名；
8. 每类聚类画像摘要；
9. 每类代表项目展示。

最终展示中，应在报告 HTML 和 Streamlit 中同时展示：

1. PCA / 聚类项目地图；
2. 聚类类型数量图；
3. 聚类方法比较表；
4. 聚类画像摘要表；
5. 各聚类类型代表项目卡片。

## 14. 机器学习模块

### 14.1 建模任务设计

系统设计两个监督学习任务：

**任务 A：评分体系代理模型。**
用于检验综合评分体系是否可由部分原始特征解释，并观察评分边界是否过度依赖少数变量。

**任务 B：项目潜力代理预测。**
用于观察原始仓库特征对项目潜力代理标签的解释能力，进一步区分高潜力项目和普通项目。

两类任务都必须进行防泄漏处理。模型训练输入不得直接使用 final_potential_score、recommendation_level、综合评分子项、risk_score 等会造成答案泄漏的字段。

### 14.2 模型方案

系统至少包括以下模型：

1. 逻辑回归；
2. 随机森林；
3. 极端随机树；
4. 梯度提升树；
5. 校准随机森林；
6. 随机基线。

输出指标包括：

1. Accuracy；
2. Precision；
3. Recall；
4. F1；
5. ROC AUC；
6. Brier Score；
7. 混淆矩阵；
8. 特征重要性；
9. 模型卡。

展示层中应将该模型解释为“表格宽深神经网络”，并在括号中保留 TabularWideDeepNet，避免长英文截断影响视觉效果。

## 17. DeepSeek 多智能体模块

系统设计 7 个 DeepSeek Agent：

1. **采集策略智能体**
   负责解释关键词、seed 仓库、多排序策略和样本覆盖范围。
2. **数据质量诊断智能体**
   负责解释缺失值、重复值、README 抓取、字段异常和样本构建质量。
3. **主题识别智能体**
   负责根据项目名称、description、topics 和 README 信号判断项目主题。
4. **评分解释智能体**
   负责解释综合潜力分、风险分和七维评分构成。
5. **个性化项目顾问智能体**
   负责根据用户画像输出 Top 项目、推荐理由、风险提示、三天复现路线和 Codex Prompt。
6. **报告生成智能体**
   负责将字段、图表、模型结果转化为结构化报告摘要。
7. **事实审查智能体**
   负责检查报告结论是否绑定字段、图表、模型指标或输出文件，避免夸大表述。

Agent 输出要求：

1. 优先调用 DeepSeek API；
2. API 失败时使用规则 fallback，但展示层不显示调试痕迹；
3. 不裸展示 JSON；
4. 不显示 source_type、user_profile、api_status 等字段名；
5. 每个 Agent 输出为自然中文解释卡片；
6. Agent 结论必须绑定真实字段、图表或模型结果；
7. Agent 输出不替代统计分析和模型指标。
